## Purpose

Drizzle ORM schema definitions (PostgreSQL table/column DDL) and database
utilities.  This package owns **all table definitions** and the migration
pipeline.

This package is **DDL-only**.  It defines tables, enums, and
migration/seed plumbing.  Query logic, repositories, and domain rules live
elsewhere (`@afenda/core` or domain-specific packages).

### What counts as "database utilities"

| âś… Allowed | âťŚ Not here |
|---|---|

| Migration runner (`migrate.ts`) | Business-rule queries (SoD, balance checks) |
| Seed helpers (`seed.ts`) | Rich query builders (â†’ `@afenda/core` or a future `@afenda/dbx`) |

> **Constrained import:** `@afenda/contracts` is allowed *only* for `*Values`
> const tuples (e.g. `InvoiceStatusValues`) to feed `pgEnum()`. These are
> plain `as const` string arrays with zero Zod runtime. Never import Zod
> schemas, entity types, or command types from contracts into this package.

### Hard enforcement (CI gate: `tools/gates/boundaries.mjs`)

1. **`*Values` only** â€” every named import from `@afenda/contracts` inside
   `packages/db/**` must end with `Values`. The boundary gate extracts
   imported identifiers and rejects anything else.
2. **No Zod** â€” `zod` is a forbidden external for `packages/db`.
3. **No deep-path imports** â€” `@afenda/contracts/src/â€¦` is rejected; import
   from the package root.
4. **Schema purity** â€” `schema/**` files must be deterministic:
   - No `process.env` reads.
   - No conditional table definitions (`if (NODE_ENV) â€¦`).
   - No runtime I/O (`fs`, `fetch`, `http`).
   - No JS-level `new Date()` defaults â€” use SQL `now()` via Drizzle's
     `.defaultNow()` or `sql\`now()\``.
5. **Barrel content** â€” `schema/index.ts` must contain only `export *`
   re-exports from local schema modules:
   - No value, type, or function definitions.
   - No external imports.

## DDL Purity Rules (`schema/**`)

Schema files must be **deterministic and side-effect free**.

| âś… Allowed | âťŚ Forbidden |
|---|---|
| `pgTable(â€¦)`, `pgEnum(â€¦)`, column helpers | `process.env` / `process.argv` usage |
| `sql\`now()\`` for timestamp defaults | `new Date()` for column defaults |
| `const tsz = (name: string) => timestamp(â€¦)` helper | `if (NODE_ENV === â€¦)` conditional DDL |
| Importing `*Values` from contracts | Any runtime I/O (`fs`, `fetch`, `http`) |
| Importing from other local `schema/*.ts` files | Importing from `@afenda/core` or `@afenda/ui` |

Rationale: Drizzle-kit reads schema files at generation time.  Side effects
or environment-dependent DDL produce non-reproducible migrations.

## Ownership Workflow (PR Review Checklist)

Reviewers **must** check the following before approving changes to
`@afenda/db`:

### New table or column

- [ ] `orgId` scoping: multi-tenant tables must have `org_id` + RLS policy
- [ ] Indexes: composite unique constraints, query-path indexes
- [ ] `createdAt` / `updatedAt` policy:
  - All tables must have at least `createdAt` (via `.defaultNow()`)
  - Mutable tables should have `updatedAt`
  - Append-only tables (audit_log, outbox_event) may omit `updatedAt`
- [ ] Soft-delete policy: if applicable, document whether `deletedAt` or
  status-based soft delete is used.  Default: hard delete unless documented.

### Infra tables (in `schema/infra.ts`)

- [ ] Retention / cleanup strategy documented:
  - `outboxEvent` â€” delivered rows pruned by worker
  - `idempotency` â€” `expiresAt` column + `cleanupExpiredIdempotency()` worker
  - `auditLog` â€” **never deleted** (append-only)
  - `sequence` â€” permanent (no cleanup)
  - `deadLetterJob` â€” manual review + periodic archive
- [ ] Unique constraints for idempotency (`org_id, command, key`), sequence
  (`org_id, entity_type, period_key`), outbox (natural key if applicable)

### Migration files

- [ ] Migration is idempotent where possible (use `IF NOT EXISTS`, etc.)
- [ ] Data migration backfills are included for new NOT NULL columns
- [ ] Migration tested locally against a seeded database
