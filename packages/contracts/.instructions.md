# Contracts Package Quality Rules

> **Auto-loaded when working with files in `packages/contracts/`**

## Hard Rules (NEVER VIOLATE)

### 1. Purity: Zod Schemas Only

**Rule:** This package contains ONLY Zod schemas and TypeScript types. No business logic, no runtime dependencies, no side effects.

```typescript
// ✅ Allowed
export const InvoiceSchema = z.object({
  id: InvoiceIdSchema,
  status: z.enum(InvoiceStatusValues),
  // ...
});

// ❌ FORBIDDEN — business logic belongs in @afenda/core
export function calculateInvoiceTotal(invoice: Invoice): bigint {
  return invoice.lines.reduce((sum, line) => sum + line.amount, 0n);
}

// ❌ FORBIDDEN — runtime dependency
import { db } from "@afenda/db";
```

**Enforcement:**

- If you write a function with business logic → move it to `@afenda/core`
- If you need database access → route through `@afenda/core` service layer
- If you need side effects → use worker handlers in `apps/worker`

### 2. JSON-First Types (No Date, No BigInt leaks)

**Rule:** All types must be safe to `JSON.stringify` across process boundaries (API → UI, API → Worker).

```typescript
// ✅ Correct
dueDate: z.string().regex(/^\d{4}-\d{2}-\d{2}$/); // ISO date string

// ❌ WRONG
dueDate: z.date(); // Date objects not JSON-safe

// ✅ Correct — bigint for money precision
amountMinor: z.coerce.bigint();

// ⚠️ Use money-transport helpers for JSON serialization
import { stringifyWithMoney, parseWithMoney } from "@afenda/contracts";
```

**Enforcement:**

- Use `UtcDateTimeSchema` for timestamps (ISO-8601 strings with `Z` suffix)
- Use `DateSchema` for business dates (`YYYY-MM-DD` strings)
- Use `z.coerce.bigint()` for money (minor units only)
- NEVER use `z.date()`, `Map`, `Set`, class instances, or functions

### 3. Import Boundaries (Zero Monorepo Dependencies)

**Rule:** Contracts can only import from `zod` and internal contract files. No cross-package imports.

```typescript
// ✅ Allowed
import { z } from "zod";
import { OrgIdSchema, PrincipalIdSchema } from "../../../shared/ids.js";

// ❌ FORBIDDEN
import { createInvoice } from "@afenda/core"; // ← core depends on contracts, not the reverse
import { invoice } from "@afenda/db"; // ← db imports from contracts, not the reverse
import { Button } from "@afenda/ui"; // ← UI is irrelevant to contracts
```

**Enforcement:**

- Imports from `@afenda/*` other than `@afenda/contracts` itself → CI failure
- Node.js runtime modules (`fs`, `path`, `crypto`, `node:*`) → CI failure
- Drizzle ORM, pg, database libraries → CI failure

### 4. Pillar Structure (shared → kernel → erp → comm)

**Rule:** New files MUST go into the correct pillar and module. No junk drawers.

**Pillar decision tree:**

```
Is this used in 3+ domains or is it IAM/infrastructure cross-cut?
  ├─ YES → shared/ (with justification comment)
  └─ NO  → Is it organization-scoped governance/identity/execution?
            ├─ YES → kernel/
            └─ NO  → Is it transactional business logic (AP/AR/GL)?
                      ├─ YES → erp/
                      └─ NO  → comm/
```

**Examples:**

- `OrgIdSchema`, `PrincipalIdSchema` → `shared/ids.ts` (IAM cross-cut)
- `InvoiceIdSchema` → `shared/ids.ts` (used in AP + GL + evidence + audit)
- `AuditLogSchema` → `kernel/governance/audit/`
- `InvoiceSchema` → `erp/finance/ap/`
- `TaskSchema` → `comm/tasks/`

### 5. File Naming Convention

**Rule:** All domain files follow these suffixes:

| Suffix          | Contains                           | Example               |
| --------------- | ---------------------------------- | --------------------- |
| `*.entity.ts`   | Canonical entity DTO schema + type | `invoice.entity.ts`   |
| `*.commands.ts` | Write/mutation command payloads    | `invoice.commands.ts` |
| `*.queries.ts`  | Read/filter/search input DTOs      | `invoice.queries.ts`  |
| `*.events.ts`   | Emitted event payloads (outbox)    | `invoice.events.ts`   |
| `index.ts`      | Domain barrel (exports only!)      | `ap/index.ts`         |

**Enforcement:**

- Non-conforming file names → suggest rename
- Logic inside `index.ts` → extract to dedicated file

### 6. Test Coverage Requirement

**Rule:** Every schema MUST have test coverage in `__vitest_test__/` directory.

```
erp/finance/ap/
  ├── invoice.entity.ts
  ├── invoice.commands.ts
  ├── __vitest_test__/
  │   ├── invoice.entity.test.ts     ← REQUIRED
  │   └── invoice.commands.test.ts   ← REQUIRED
  └── index.ts
```

**Test template:**

```typescript
import { describe, it, expect } from "vitest";
import { InvoiceSchema, InvoiceStatusValues } from "../invoice.entity.js";

describe("InvoiceSchema", () => {
  it("should validate a complete invoice", () => {
    const result = InvoiceSchema.safeParse({ /* valid data */ });
    expect(result.success).toBe(true);
  });

  it("should reject invalid status", () => {
    const result = InvoiceSchema.safeParse({ /* invalid status */ });
    expect(result.success).toBe(false);
  });
});

describe("InvoiceStatusValues", () => {
  it("should be stable (breaking change detector)", () => {
    expect(InvoiceStatusValues).toEqual(["draft", "submitted", "approved", "posted", "paid", "rejected", "voided"]);
  });
});
```

**Enforcement:**

- Missing `__vitest_test__/` directory → create it
- Missing test file for schema → create it
- Coverage < 90% → add tests

### 7. Barrel Exports Only

**Rule:** `index.ts` files contain ONLY `export * from ...` statements. No logic, no types, no schemas, no comments beyond header.

```typescript
// ✅ Correct
/**
 * Barrel — erp/finance/ap (accounts payable)
 */
export * from "./invoice.entity.js";
export * from "./invoice.commands.js";
export * from "./payment-terms.entity.js";

// ❌ WRONG — type defined in barrel
export type InvoiceWithLines = Invoice & { lines: InvoiceLine[] };

// ❌ WRONG — helper function in barrel
export function isApproved(invoice: Invoice): boolean {
  return invoice.status === "approved";
}
```

**Enforcement:**

- Non-export statements in barrel → extract to dedicated file
- Barrel > 60 lines → split into sub-module barrels

### 8. Shared/ Escalation Criteria

**Rule:** Primitives go in `shared/` ONLY if they meet **one of two criteria**:

1. **Cross-domain (3+ modules):** Referenced in 3 or more domain modules
2. **Infrastructure cross-cut:** IAM, correlation, headers, envelopes, error codes

**Justification comment required:**

```typescript
// cross-domain: referenced by invoice, gl (journal lines), evidence, audit
export const InvoiceIdSchema = uuid.brand<"InvoiceId">();
```

**Enforcement:**

- New ID in shared/ without justification → ask "Is this used in 3+ domains?"
- Domain-specific primitive in shared/ → move to domain folder
- Unjustified escalation → move back to domain folder

## Common Quality Issues to Detect

### Issue: Business Logic in Contracts

**Detection:**

- Functions with `if`/`switch` logic
- Methods that mutate state
- Functions with side effects (DB, HTTP, file I/O)

**Resolution:**

- Move to `packages/core/src/<pillar>/<module>/`
- Keep only pure Zod schemas and types in contracts

### Issue: Runtime Dependencies

**Detection:**

- Imports from `@afenda/db`, `@afenda/core`, `@afenda/ui`
- Imports from `drizzle-orm`, `pg`, Node.js modules

**Resolution:**

- Remove import
- Restructure to use proper layering

### Issue: Missing Tests

**Detection:**

- Schema file exists but no corresponding test file in `__vitest_test__/`

**Resolution:**

- Create `__vitest_test__/<filename>.test.ts`
- Add tests for valid cases, invalid cases, refinements, status values

### Issue: ID Over-Escalation

**Detection:**

- ID in `shared/ids.ts` without cross-domain justification comment
- ID only used in one domain

**Resolution:**

- Move to domain file: `export const WidgetIdSchema = brandedUuid("WidgetId");`
- Import from `@afenda/contracts` (brandedUuid helper)

### Issue: Non-JSON-Serializable Types

**Detection:**

- `z.date()` usage
- `BigInt` outside money contexts
- `Map`, `Set`, class instances

**Resolution:**

- Replace `z.date()` with `UtcDateTimeSchema` or `DateSchema`
- Keep `BigInt` only for money (with `z.coerce.bigint()`)
- Use plain objects and arrays

### Issue: Logic in Barrel Files

**Detection:**

- Type definitions in `index.ts`
- Helper functions in `index.ts`
- More than just `export * from ...`

**Resolution:**

- Extract to dedicated file (e.g., `types.ts`, `helpers.ts`)
- Keep barrel pure

## Development Workflow

### Adding a New Entity

1. **Choose pillar/module** (see decision tree above)
2. **Create entity file:** `<entity>.entity.ts`
   - Define `*Values` arrays as `as const`
   - Define Zod schema with `z.object(...)`
   - Export inferred type: `export type Entity = z.infer<typeof EntitySchema>;`
3. **Create command file:** `<entity>.commands.ts`
   - Every command MUST include `idempotencyKey: IdempotencyKeySchema`
4. **Create test file:** `__vitest_test__/<entity>.entity.test.ts`
5. **Update barrel:** Add to `index.ts` in module directory
6. **Update OWNERS.md:** Document new exports
7. **Run checks:**
   ```bash
   pnpm typecheck
   pnpm test
   pnpm check:all
   ```

### Refactoring Existing Schema

1. **Read current schema** — understand intent, usage, refinements
2. **Check dependents** — grep for imports across monorepo
3. **Update schema** — maintain backward compatibility or document breaking change
4. **Update tests** — ensure all cases still pass
5. **Update OWNERS.md** — if exports changed
6. **Run checks:**
   ```bash
   pnpm typecheck
   pnpm test
   pnpm check:all
   ```

### Moving Files Between Pillars

1. **Verify justification** — Does it meet escalation criteria?
2. **Update imports** — across all dependent packages
3. **Update barrel exports** — both old and new locations
4. **Update OWNERS.md** — in both old and new locations
5. **Run checks:**
   ```bash
   pnpm typecheck
   pnpm test
   pnpm check:all
   ```

## Pre-Commit Checklist

Before committing changes to `packages/contracts/`:

- [ ] All new schemas have corresponding test files
- [ ] Test coverage > 90% for changed files
- [ ] No imports from `@afenda/db`, `@afenda/core`, `@afenda/ui`
- [ ] No `z.date()`, only `UtcDateTimeSchema` or `DateSchema`
- [ ] No business logic (functions with conditionals, side effects)
- [ ] Barrel files contain only `export * from ...`
- [ ] New IDs in `shared/ids.ts` have justification comments
- [ ] OWNERS.md updated with new exports
- [ ] `pnpm typecheck` passes
- [ ] `pnpm test` passes
- [ ] `pnpm check:all` passes (all 18 CI gates)

## When to Escalate

**Ask for review if:**

- Adding a new pillar or module
- Escalating a primitive to `shared/` (needs justification)
- Introducing a new pattern not documented here
- Breaking backward compatibility in a schema
- Removing or renaming an error code, permission, or audit action
