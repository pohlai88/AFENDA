Core principle

The 4-layer dark system

Depth should communicate spatial hierarchy before borders or shadows do.
L0 · Atmosphere

Background / recess / outer shell

Use for the app frame, page backdrop, and negative space. It should never fight with content.

  • Tokens: --surface-l0, --background
  • Use for shell canvas, gutters, body background
  • Never place dense UI controls directly here
L1 · Operating surfaces

Work area / page plane

Use for the main working canvas. This is where dashboards, forms, and data modules live.

  • Tokens: --surface-l1, --surface-100–--surface-250
  • Best for sections, shells, fixed page areas
  • Stable, quiet, low-contrast depth
L2 · Elevated content

Cards / widgets / important blocks

Use for KPI cards, filters, inspectors, cards, and embedded modules that need lift.

  • Tokens: --surface-l2, --card, --surface-275–--surface-350
  • Good for card stacks and compositional grouping
  • Depth comes from surface first, shadow second
L3 · Floating context

Popover / command / dropdown

Use for transient UI. Popovers, drawers, sheets. Pair with --scrim when overlaying.

  • Tokens: --surface-l3, --popover, --sheet, --drawer
L4 · Modal authority

Modal / critical overlay

Highest Z. Use for modals and blocking dialogs. Always pair with --scrim and --shadow-modal.

  • Tokens: --surface-l4, --modal, --tooltip

Token utilization

Where each token family should be used

1. Surface tokens

  • Use surface tokens for spatial depth only.
  • Move between L0 → L4 to express importance and containment.
  • Do not tint every card with semantic meaning.
  • Most ERP screens should live between L1 and L2.

2. Semantic lane tokens

  • --primary for truth, product identity, major CTA.
  • --interactive for operator action and navigation intent.
  • --premium for governance, institutional, high-trust moments.
  • Keep one dominant lane per component.

3. Status tokens

  • Use status only for real workflow state.
  • Never reuse chart palette as workflow semantics.
  • Status backgrounds should be soft, not neon.
  • Text must remain readable in dense grid cells.

Usage guideline

Use chart tokens for comparison, series differentiation, thresholds, heatmaps, and scenario lines.

Do not use status colors as arbitrary chart colors because users will misread them as workflow state.

Reserve red/green semantics for cases where positive/negative is truly meaningful, such as variances or gains/losses.

Implementation guide

How to apply the tokens in product design

Recommended mapping

App shell: --background, --sidebar, --topbar

Page sections: --surface-100 to --surface-200

Cards & widgets: --card, --surface-275 to --surface-350

Floating UI: --popover, --sheet, --modal, --scrim

Inputs: --field-bg, --field-border, --ring

Dense tables: --grid, --grid-border, --grid-header, --row-even

Rules of discipline

1. Let surface hierarchy solve layout depth before adding glow or border tricks.

2. Keep tertiary text readable; graphite mode fails when secondary text becomes fog.

3. Reserve premium gold for governance and trust, not generic emphasis.

4. Use focus ring consistently across forms, tables, and command actions.

5. Most screens should feel 80% neutral, 15% semantic, 5% dramatic.
